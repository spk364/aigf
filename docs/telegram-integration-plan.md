# Telegram Integration Plan

Цель: дать пользователям GFAI возможность общаться со своими персонажами в Telegram — нативная переписка в чате, единая история и биллинг с веб-приложением.

## Решения по архитектуре (стартовая ставка)

| Развилка | Выбор | Почему |
|---|---|---|
| Сколько ботов | **Один бот** `@gfai_bot` с выбором персонажа | Один токен/деплой/модерация. Risk одного бана меньше N. Можно эволюционировать в гибрид позже. |
| Транспорт | **Bot API + Mini App (гибрид)** | Чат — нативный (вся фишка). Mini App — для биллинга, галереи, настроек, профиля персонажа. |
| Линковка | **Deep-link OTT** | `/start` в TG → одноразовый токен → `gfai.app/link?t=...` → связь `telegramId ↔ userId` в Payload. |
| Платежи | **Telegram Stars + fallback на веб-чекаут** | Stars нативны, обходят 30% iOS. Веб — для крупных пакетов и подписок. |
| Адаптер AI | **Тот же пайплайн что и на сайте** | Чат, генерация изображений, голос — переиспользуем существующие сервисы (включая fal.ai endpoints). |

## Этапы

### Этап 0 — Discovery / risk gate (1–2 дня)
**Не пишем код, пока не закроем:**
- [ ] Политика Telegram по adult-ботам в 2026: что именно бан-фактор (явная порнография в media vs. NSFW-текст vs. имя бота).
- [ ] Возрастной gate: достаточно ли disclaimer'а на `/start` или нужен внешний KYC/age-verify (зависит от юрисдикции).
- [ ] Telegram Stars: текущая комиссия, payout-политика, поддержка возвратов.
- [ ] CSAM-риск: входящие фото от пользователя — нужен NSFW + age-классификатор на входе. Решить: разрешаем ли вообще приём фото в MVP.
- [ ] Юридический ревью: TOS GFAI должны явно покрывать TG-канал.

**Output:** короткий risk-memo + go/no-go на конкретные фичи (приём фото, voice, push-инициатива).

### Этап 1 — MVP бота (≈ 1.5 недели)
**Цель:** один пользователь может залогиниться, выбрать персонажа и переписываться текстом.

- [ ] Создать `@gfai_bot` через BotFather, токен в Vercel env (staging + prod отдельные боты).
- [ ] Webhook endpoint в Next.js: `app/api/telegram/webhook/route.ts` (POST, проверка `secret_token` по заголовку TG).
- [ ] Библиотека: **grammY** (легковесная, TS-first, лучше Telegraf для serverless).
- [ ] Команды:
  - `/start [linkToken?]` — приветствие, age-gate (inline-кнопка "Мне 18+"), линковка если есть токен.
  - `/link` — генерит OTT в Payload, отдаёт URL `gfai.app/link?t=...`.
  - `/companions` — inline-список персонажей пользователя (`character_id` в callback_data).
  - `/switch` — сменить активного персонажа.
  - `/balance` — текущие токены, кнопка "Купить" → Mini App.
  - `/reset` — очистить контекст текущего диалога (с подтверждением).
- [ ] Payload collection `TelegramAccounts`: `{ telegramId, userId, currentCharacterId, linkedAt, lastSeenAt }`.
- [ ] Endpoint `app/link/page.tsx`: принимает токен, требует логин, делает связку, помечает токен использованным.
- [ ] Текстовый чат: входящее сообщение → достаём `currentCharacterId` → пушим в существующий chat-сервис → стримим ответ через `sendChatAction("typing")` каждые 4 сек, отправляем итоговое сообщение (или несколько, если длинное).
- [ ] Списание токенов из той же квоты что и веб (одна правда в Payload).
- [ ] Rate limit на пользователя (например 1 запрос / 2 сек).
- [ ] Логирование диалогов в ту же таблицу что и веб (с `source: "telegram"`).

**DoD:** залогиненный пользователь может в TG обменяться 20 сообщениями с персонажем, история видна на сайте, токены списываются.

### Этап 2 — Mini App + Stars (≈ 1 неделя)
- [ ] Mini App URL зарегистрировать через BotFather (`/setmenubutton`).
- [ ] Reuse существующий Next.js UI: страница `/tg` с TG WebApp SDK, авто-логин по `initData` (HMAC-валидация на сервере).
- [ ] Экраны MVP в Mini App: баланс, галерея персонажа, профиль, покупка пакетов.
- [ ] Telegram Stars invoice: `sendInvoice` с XTR currency, webhook на `pre_checkout_query` + `successful_payment` → пополнение баланса в Payload.
- [ ] Inline-кнопка "💎 Купить токены" в боте → открывает Mini App на экране покупки.

**DoD:** пользователь может купить пакет токенов через Stars не выходя из TG.

### Этап 3 — Медиа (≈ 1 неделя)
- [ ] Команда `/photo <prompt>` или просто текст с триггером "пришли фото" → генерация через тот же fal.ai пайплайн → `sendPhoto`.
- [ ] Списание токенов по той же тарификации.
- [ ] **Решение discovery:** разрешён ли приём фото от пользователя. Если да — NSFW-классификатор + age-классификатор на входе, при срабатывании отклоняем без сохранения.
- [ ] Голос: входящие voice messages → STT (Whisper или существующий пайплайн) → ответ текстом + опционально TTS через `sendVoice`.

**DoD:** пользователь может попросить фото и получить его; голосовые работают в обе стороны (если решим включить).

### Этап 4 — Retention (≈ 1 неделя, опционально)
**Push-инициатива от персонажа** — мощный retention, но spam-риск.
- [ ] Strict opt-in: на `/start` отдельная кнопка "Разреши Арии писать первой".
- [ ] Per-character rate limit: не чаще 1 в 24ч, не в quiet hours юзера.
- [ ] Триггеры: N дней молчания, апдейты галереи, события в сюжете персонажа.
- [ ] Лёгкая команда отписки: `/quiet` → пауза на 7 дней, `/quiet off` → возврат.

**DoD:** retention D7 на TG-юзерах ≥ retention D7 на веб-юзерах, spam-репортов нет.

### Этап 5 — Гибрид с именными ботами (опционально, после product-fit)
Только когда MVP устаканится.
- [ ] Топ-3 персонажа получают свои `@aria_gfai_bot` etc.
- [ ] Один общий webhook routing по `bot_id` → знает какой это персонаж.
- [ ] Новые регистрации идут через основной бот, "именные" — для уже знакомых пользователей.

## Технические заметки

- **Webhook vs long polling:** на Vercel — только webhook (нет персистентного процесса). Secret token в заголовке обязателен.
- **Idempotency:** TG может ретраить апдейт. Хранить `update_id` в Redis с TTL 5 мин, дедуплицировать.
- **Длинные ответы:** TG лимит 4096 символов. Резать по предложениям, не по байтам.
- **Стриминг:** в TG нет true streaming. Шлём `typing...` через `sendChatAction` каждые 4 сек пока модель пишет, потом одно итоговое сообщение. Альтернатива — слать частями и редактировать (`editMessageText`), но шумно в чате.
- **Отдельные боты для staging/prod:** чтобы не путать webhooks. Domain check в middleware.
- **Метрики:** D1/D7/D30, среднее число сообщений в день, conversion в покупку Stars, отказы по 18+.
